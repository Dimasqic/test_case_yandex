Задание 1. Тест-кейсы на переход к странице поиска

Предусловия для всех тест-кейсов:
•	Открыт веб-браузер на персональном компьютере.
•	Выполнен переход на главную страницу по адресу: https://ya.ru.

Тест-кейс №1: Успешный переход к результатам поиска при клике на кнопку со стрелкой
Шаги:
•	Нажать левой кнопкой мыши на поисковую строку.
•	Ввести валидный текстовый запрос (например, «погода»).
•	Кликнуть левой кнопкой мыши по кнопке поиска (красная кнопка со стрелкой «>» справа в поисковой строке).
Ожидаемый результат: Происходит перенаправление на страницу поисковой выдачи Яндекс (URL содержит /search/) с результатами по запросу «погода». Поисковая строка на новой странице заполнена введенным текстом.

Тест-кейс №2: Успешный переход к результатам поиска с помощью клавиатуры (Enter)
Шаги:
•	Нажать левой кнопкой мыши на поисковую строку.
•	Ввести валидный текстовый запрос (например, «погода»).
•	Нажать клавишу Enter на клавиатуре.
Ожидаемый результат: Происходит перенаправление на страницу поисковой выдачи Яндекс с результатами по запросу «погода». Страница открывается корректно, элементы интерфейса стабильны.

Тест-кейс №3: Успешный переход к результатам поиска при выборе поисковой подсказки
Шаги:
•	Нажать левой кнопкой мыши на поисковую строку.
•	Ввести начальную часть запроса (например, «пого»).
•	Дождаться появления выпадающего списка подсказок (саджеста).
•	Кликнуть левой кнопкой мыши по первой подсказке в списке (например, «погода волгоград»).
Ожидаемый результат: Происходит перенаправление на страницу поисковой выдачи Яндекс с результатами по запросу «погода волгоград». В поисковой строке отображается текст выбранной подсказки.



Задание 2. Исследовательское тестирование бесконечной ленты на ya.ru
1. Продуктовый риск: Отсутствие кнопки быстрого возврата «Наверх»
•	Наблюдение: При длительном скроллинге ленты пользователь уходит далеко вниз от первоначального экрана. Фиксированная кнопка возврата к началу страницы отсутствует.
•	Аргументация: Главная функция ya.ru — строка поиска. Если пользователю в процессе чтения резко понадобится поиск, ему придется долго скроллить обратно вручную. Это ухудшает UX и снижает конверсию в поиск.
2. Вопрос к логике / UX: Хаотичная «мозаичная» структура сетки и проблема доступности (Accessibility)

•	Наблюдение: Лента использует сетку Masonry, где карточки имеют разную высоту. При перемещении по карточкам с помощью клавиши Tab фокус прыгает хаотично вверх и вниз, а сама лента не прокручивается автоматически.
•	Аргументация: Нарушение стандартов доступности интерфейсов (WCAG). Взгляд пользователя «мечется» по экрану. Для инклюзивной аудитории без мыши интерфейс становится недоступен. Необходимо выровнять сетку для корректной навигации клавиатурой.
3. Технический риск: Высокий порог занимаемой памяти (Memory Consumption)
•	Наблюдение: Вкладка в браузере (Chromium/Edge) изначально потребляет около 415 МБ ОЗУ. При активном автоскролле за 30 секунд потребление оперативной памяти возрастает до 1000–1200 МБ и фиксируется на этом верхнем лимите. Общая загрузка ОЗУ системы доходит до 70%.
•	Аргументация: Тяжелые элементы подгружаются в DOM без удаления старых блоков. Выделение более 1 ГБ под одну вкладку критично для слабых устройств (с 8 ГБ ОЗУ). Это вызывает лаги ОС или падение страницы с ошибкой Out of Memory. Требуется оптимизировать кэширование.
4. Вопрос к логике: Неочевидный статус взаимодействия («Лайк») в режиме инкогнито
•	Наблюдение: Неавторизованный пользователь может нажать «Лайк» на карточке. Кнопка окрашивается, но счетчик лайков не отображается, и система не предлагает войти в аккаунт.
•	Аргументация: Логика действия скрыта. Пользователь не понимает, влияет ли лайк на рекомендации в текущей сессии или уйдет в никуда после закрытия вкладки. Рекомендуется выводить поп-ап авторизации или показывать счетчик для видимого результата.
5. Вопрос к логике / UX: Принудительная пауза медиаплеера при потере фокуса вкладки
•	Наблюдение: Переключение на другую вкладку ставит видео в ленте на паузу. При возвращении фокуса ролик автоматически продолжается с того же места.
•	Аргументация: Если видео содержит полезный разговорный контент (например, рецепт с товаром Маркета), пользователь может захотеть слушать его фоном. Принудительный стоп ломает этот сценарий. Стоит уточнить требования и добавить опцию фонового аудио.
6. Предложение по улучшению (Feature Request): Отсутствие перемотки и ускорения видео
•	Наблюдение: Встроенный плеер не имеет ползунка временной шкалы (таймлайна) для ручной перемотки и функции изменения скорости (1.5х / 2х).
•	Аргументация: Риск снижения удержания аудитории (Retention). Современные пользователи привыкли к быстрому потреблению информации (TikTok/Shorts). Без перемотки длинные коммерческие ролики (обзоры товаров) будут чаще закрывать, снижая конверсию в покупки.
7. Бизнес-логика / UX: Избыточная плотность и ротация рекламных блоков (Ad Density)
•	Наблюдение: При скролле два разных рекламных блока могут отображаться подряд в соседних ячейках одного горизонтального ряда.
•	Аргументация: Высокая плотность рекламы подряд вызывает ощущение спама и снижает лояльность к сервису. Рекламные карточки конкурируют друг с другом, снижая кликабельность (CTR) обеих. Необходим жесткий шаг ротации (например, 1 блок на 5 карточек контента).
